# 노리(Nori) MVP 개발 계획

> 우리 아이 놀이 친구 AI — 발달 단계별 맞춤 놀이 추천 · 놀이 방법 안내 · 준비물 구매 연결 · 음성 인터페이스

이 문서는 '모두의 창업' MVP 완성과 베타 검증까지의 단계별 개발 계획입니다.
각 Phase는 **독립적으로 배포 가능한 단위**로 설계되어, 단계마다 동작하는 결과물을 확인하며 진행합니다.

---

## 제품 원칙 (모든 Phase에 적용)

1. **비교 없는 양육**: 'N개월 평균' 식의 또래 비교 수치를 어떤 화면에도 노출하지 않습니다. 오직 내 아이의 현재 상태와 다음 놀이에만 집중합니다.
2. **진단 금지, 안내만**: AI는 발달 지연 여부를 추측·판단하지 않습니다. 위험 신호(Red Flag) 키워드 감지 시 AI 생성 답변을 차단하고, 사전 작성된 '전문가 상담 권고' 안내만 노출합니다(의료법 리스크 차단).
3. **핸즈프리 우선**: 부모의 손이 자유롭도록 음성 입력과 음성 안내를 1급 시민으로 설계합니다.
4. **탐색→구매 원스톱**: 놀이 레시피마다 준비물 구매 링크(아웃바운드)를 함께 제공합니다.

## Phase 2 — 자녀 프로필 & 온보딩 & 홈 대시보드

**목표**: 로그인 후 자녀 정보를 등록하고, 홈에서 "○○이(24개월)를 위한 오늘의 놀이"를 볼 수 있는 기반을 만든다.

### DB 스키마 (Supabase 마이그레이션)

```sql
create table children (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  name text not null,                -- 애칭 (예: 콩이)
  birth_date date not null,          -- 개월수는 저장하지 않고 매번 계산
  interests text[] default '{}',     -- 관심사 태그 (자동차, 그리기, 물놀이 …)
  notes text,                        -- 발달 특이사항 자유 입력 (선택)
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
alter table children enable row level security;
-- RLS: 본인 자녀만 select/insert/update/delete
```

### 작업 항목

- [x] `children` 테이블 마이그레이션 + RLS 정책 4종 (`supabase/migrations/0001_children.sql`)
- [x] 온보딩 플로우 `/onboarding` (자녀 미등록 시 `/home`에서 리다이렉트)
  - 애칭·생년월일·관심사 태그·특이사항을 한 화면에서 입력 (MVP는 단일 폼으로 단순화)
- [x] 자녀 프로필 CRUD (`/children`, `/children/[id]/edit`, 다자녀 지원 — 활성 자녀 쿠키 전환)
- [x] 홈 대시보드 개편: 활성 자녀(애칭·개월수), 자녀 전환 칩, "노리에게 물어보기" 진입 버튼(Phase 3까지 비활성)
- [x] `lib/age.ts`: 생년월일 → 개월수 계산 유틸 (생후 N개월 / 만 X세 표기)

### 완료 기준

- [x] 신규 가입자가 온보딩을 마치면 홈에서 활성 자녀 정보가 보인다.
- [x] RLS 정책으로 본인 자녀만 select/insert/update/delete 가능 (마이그레이션에 정의).

> 비개발자 메모: 위 마이그레이션 SQL은 Supabase 대시보드의 **SQL Editor**에 붙여넣어 실행하거나
> `supabase db push`로 적용하세요. 적용 후 앱에서 온보딩이 동작합니다.

## Phase 3 — 채팅 UI & 음성 인터페이스 (목업 응답)

**목표**: "노리야, 종이컵으로 할 수 있는 놀이 찾아줘"를 음성/텍스트로 입력하고 답을 받는 대화 화면을 완성한다. AI 연동 전이므로 응답은 고정 샘플 레시피로 대체한다.

### 작업 항목

- [ ] `/chat` 대화 화면: 말풍선 리스트, 입력창, 전송
- [ ] 음성 입력(STT): Web Speech API `SpeechRecognition` (ko-KR)
  - 마이크 버튼 길게/탭 → 청취 시작, 인식 텍스트 입력창에 반영
  - 미지원 브라우저(Firefox 등) 감지 시 텍스트 입력으로 자연스럽게 폴백
- [ ] 음성 안내(TTS): `speechSynthesis`로 레시피 요약 읽어주기 + 일시정지/정지
- [ ] 레시피 응답 카드 컴포넌트 v1: 놀이 제목 / 발달 포인트 / 준비물 / 순서 / 주의사항
- [ ] 목업 응답기 `lib/mock-recipes.ts`: 키워드(종이컵, 물놀이 등)별 샘플 3~5종

### 완료 기준

- 스마트폰 Chrome/Safari에서 음성으로 질문 → 샘플 레시피 카드 + 음성 안내까지 동작.
- Web Speech 미지원 환경에서도 텍스트로 동일 흐름 사용 가능.

### 리스크

- iOS Safari의 STT 제약(권한, 백그라운드 중단)이 있으므로 **실기기 테스트를 이 단계에서 반드시 수행**. 품질이 부족하면 Phase 6 이후 전문 STT(API)로 교체 여지를 인터페이스로 분리해 둔다 (`lib/speech/` 추상화).

## Phase 4 — Gemini 놀이 레시피 엔진 + Red Flag 안전 가드

**목표**: 목업을 실제 AI로 교체한다. 자녀 프로필(개월수·관심사)을 컨텍스트로 맞춤 레시피를 생성하되, 안전 가드를 같은 단계에서 함께 구축한다(안전은 후순위가 아님).

### 아키텍처

```
클라이언트 → POST /api/chat (Route Handler, 서버 전용 GEMINI_API_KEY)
  1. 입력 전처리: Red Flag 키워드 검사 (서버, AI 호출 전)
     └ 감지 시: AI 호출 없이 { type: "red_flag" } 반환 → 하드코딩 안내 컴포넌트 렌더
  2. 통과 시: 시스템 프롬프트(자녀 개월수·관심사·제품 원칙) + Gemini 호출
  3. 구조화 출력(JSON): { title, devPoints[], materials[], steps[], cautions[] }
  4. 출력 후처리: 진단성 표현 필터 (이중 안전망)
```

### 작업 항목

- [ ] `app/api/chat/route.ts`: Gemini 호출 Route Handler (스트리밍은 v2, 우선 단건 응답)
- [ ] 시스템 프롬프트 설계: 비교 금지·진단 금지·놀이 가이드 역할 고정, 개월수별 안전 수칙(삼킴 주의 등) 포함
- [ ] 구조화 출력 스키마 + Zod 검증 (`lib/recipe-schema.ts`)
- [ ] **Red Flag 가드** (`lib/red-flag.ts`)
  - 키워드 사전: 발달 지연 의심 표현(눈맞춤 안 함, 호명 반응 없음, 옹알이 없음, 퇴행 등)
  - 감지 시 AI 미호출 + `<RedFlagNotice />` 렌더: 따뜻한 어조의 사전 작성 문구 + 전문가 상담 권고(소아청소년과, 발달센터, 육아종합지원센터 안내)
  - 키워드 사전은 코드와 분리된 데이터 파일로 관리해 비개발자도 검수 가능하게
- [ ] 대화 기록 저장: `conversations` / `messages` 테이블 (RLS 포함) — 레시피 재열람의 기반
- [ ] 에러/한도 처리: API 실패 시 친절한 재시도 안내, 사용자별 일일 호출 제한(rate limit)

### 완료 기준

- "26개월, 자동차 좋아하는 아이랑 뭐하고 놀까?" → 개월수·관심사가 반영된 레시피 생성.
- Red Flag 문장 10종 테스트 케이스 전부에서 AI 응답이 차단되고 안내 컴포넌트가 뜬다.
- GEMINI_API_KEY가 클라이언트 번들에 노출되지 않는다.

## Phase 5 — 레시피 고도화: 영상 가이드 · 커머스 링크 · 저장함

**목표**: 레시피 카드를 "보고 따라 하고 바로 준비"가 가능한 완성형으로 만든다. 수익 모델(제휴 링크)의 기술 기반을 깐다.

### 작업 항목

- [ ] **영상 가이드**: 레시피 ↔ 직접 제작 YouTube 영상 매핑 테이블(`recipe_videos`), 카드 내 임베드 재생
  - 영상이 없는 레시피는 영역 자체를 숨김 (빈 박스 노출 금지)
- [ ] **커머스 링크**: 준비물별 구매 링크 데이터 관리 (`materials` 마스터 테이블: 이름, 구매 URL, 제휴 여부)
  - 1차는 쿠팡 파트너스 등 아웃바운드 링크. "광고/제휴 링크 포함" 고지 문구 필수
  - 클릭 이벤트 로깅 → 전환율 분석의 기초 데이터
- [ ] **저장함**: 레시피 북마크(`saved_recipes`), `/library`에서 다시 보기 + 대화 히스토리
- [ ] 홈 대시보드에 "오늘의 추천 놀이" 1건 자동 노출 (개월수 기반)

### 완료 기준

- 레시피에서 영상 시청 → 준비물 링크 클릭 → 외부 구매 페이지 이동이 한 화면에서 이뤄진다.
- 저장한 레시피가 재로그인 후에도 유지된다.

**목표**: Vercel 프로덕션 배포, 컴플라이언스 점검, 맘카페 베타 테스터 모집을 위한 준비를 마친다.

### 작업 항목

- [ ] Vercel 배포 + 커스텀 도메인, Supabase 프로덕션 환경 분리(redirect URL 등)
- [ ] **컴플라이언스 점검** (멘토링 결과 반영)
  - 서비스 전역 고지: "노리는 의료적 진단·치료를 제공하지 않으며 놀이 가이드만 제공합니다"
  - 이용약관·개인정보처리방침 (자녀 생년월일 = 민감할 수 있는 개인정보 → 수집 최소화 원칙 명시)
  - 제휴 링크 광고 표기 점검
- [ ] 기본 분석: 페이지뷰, 질문 수, 레시피 생성 수, 커머스 클릭 수 (Vercel Analytics or PostHog)

- [ ] 베타 온보딩 자료: 사용법 숏츠 1편 + 맘카페 모집 공지문

### 완료 기준

- 외부 사용자가 URL 접속 → 가입 → 자녀 등록 → 음성 질문 → 레시피 수신 → 구매 링크 클릭까지 전 과정을 막힘없이 완료한다.

## Phase 별 의존 관계와 순서 근거

- Phase 2가 가장 먼저인 이유: 자녀 프로필(개월수·관심사)이 AI 맞춤화의 입력값이므로, 데이터 모델이 모든 후속 기능의 토대.
- Phase 3에서 AI 없이 UI를 먼저 만드는 이유: 음성 UX는 실기기 검증 리스크가 가장 크므로 빨리 부딪혀 보고, AI 비용 없이 화면 흐름을 확정.
- Phase 4에서 Red Flag를 같이 만드는 이유: 안전 가드 없는 AI를 단 하루도 노출하지 않기 위함 (컴플라이언스 원칙).
- Phase 5의 커머스가 늦은 이유: 레시피 품질이 먼저 검증돼야 구매 전환이 의미를 가짐.

## MVP 이후 백로그 (이번 범위 제외)

- 스트리밍 응답 + 전문 STT/TTS 교체 (음성 품질 고도화)
- 발달 영역별(대근육·소근육·언어·사회성) 놀이 기록·회고
- 푸시 알림 (오늘의 놀이 리마인드), PWA 설치
- 제휴 커머스 정식 계약 및 인앱 결제 검토
- 전문가(아동발달) 콘텐츠 감수 체계
